## 1. 项目现状

### 1.1 当前基础

| 维度 | 现状 |

| **当前发布渠道** | [Hangar](https://hangar.papermc.io/OrzMC/OrzMC)（免费）、[Modrinth](https://modrinth.com/plugin/orzmc)（免费） |
| **发布方式** | CI/CD 全自动化（GitHub Actions），Push main → Snapshot/Beta，Tag → Release |

| **用户群** | 私服 `orzmc.jokerhub.cn` 实际运行（dogfooding） |

### 1.2 商业化的核心优势

- **功能成熟** — 不是玩具项目，有完整测试和架构
- **多平台机器人集成** — QQ + Discord + 飞书，**极少有插件同时支持三者**
- **中文市场独特性** — 海外插件没有 QQ/飞书集成，这是你的独占蓝海
- **代码质量好** — 分层架构、测试覆盖、文档健全，商业化后维护成本可控

### 1.4 竞争壁垒（护城河）分析

商业化成功的关键不仅是"有好功能"，更是"有竞品难以复制的壁垒"。以下是 OrzMC 的护城河评估：

#### 核心壁垒

| 壁垒 | 强度 | 说明 |
|------|:----:|------|
| **QQ 机器人集成** | 🟢 强 | 海外竞品**不可能**做——没有 QQ 生态认知、没有 NapCatQQ/OneBot 对接经验。对中国服主来说，这是选择 OrzMC 的决定性理由。 |
| **三平台统一** | 🟢 强 | QQ + Discord + 飞书的组合在 MC 插件市场中是**唯一的**。竞品要做需要同时熟悉三种协议，开发成本极高。 |
| **代码质量 + 架构** | 🟡 中 | 六边形架构、测试覆盖、CI/CD —— 降低了长期维护成本，但竞品也可以通过重构达到。 |
| **中文原生支持** | 🟡 中 | 中文文档、中文错误提示、中国开发者直接支持。海外竞品不愿投入中文化。 |
| **先发优势 + dogfooding** | 🟡 中 | 已有实际运行的私服验证稳定性。新进入者需要时间积累 trust。 |
| **开源社区** | 🟡 中 | GPLv3 开源降低了采纳门槛，社区可以贡献。但如果竞品 fork 并改进，也可能成为威胁。 |

#### 薄弱环节

| 薄弱点 | 严重程度 | 应对 |
|--------|:--------:|------|
| **单一开发者** | 🔴 高 | 所有代码一人维护，"巴士因子"=1。如果停止维护，项目即死。→ 开源 + 降低社区贡献门槛可部分缓解。 |
| **功能可复制性** | 🟡 中 | 白名单管理、TNT 保护等单一功能任何开发者都能写。→ 核心壁垒在于功能组合 + QQ 集成，而非单项功能。 |
| **依赖第三方平台** | 🟡 中 | NapCatQQ 不受你控制。→ 关注上游，必要时自研 QQ 适配层。 |
| **缺乏品牌认知** | 🟡 中 | OrzMC 目前没有品牌影响力。→ 通过 B站/论坛内容营销 + 用户口碑逐步建立。 |

#### 与主要竞品的差异化矩阵

| 功能维度 | OrzMC | EssentialsX | CMI | LuckPerms | 典型小插件 |
|---------|:-----:|:-----------:|:---:|:---------:|:---------:|
| QQ Bot | ✅ | ❌ | ❌ | ❌ | ❌ |
| Discord Bot | ✅ | ❌ | ❌ | ❌ | 部分 |
| 飞书通知 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 白名单管理 | ✅ | ❌ | ❌ | ❌ | ✅ 仅此功能 |
| 跨服传送门 | ✅ | ❌ | ✅ | ❌ | ❌ |
| TNT 保护 | ✅ | ❌ | ✅ | ❌ | ✅ 仅此功能 |
| GeoIP 过滤 | ✅ | ❌ | ❌ | ❌ | ✅ 仅此功能 |
| 世界备份/优化 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 运行时配置 | ✅ | ❌ | ✅ | ❌ | ❌ |
| 开源免费 | GPLv3 | GPLv3 | 付费 | MIT | 各异 |
| 中文化 | ✅ 原生 | 翻译文件 | 翻译文件 | 翻译文件 | 部分 |

> **关键洞察**：没有任何单一竞品覆盖 OrzMC 的全功能矩阵。OrzMC 本质上是 "QQ Bot 运维面板 + 综合管理套件" 的**套餐捆绑**（bundling）策略。用户安装 OrzMC 一个插件就能取代 3-5 个独立插件，降低运维复杂度——这是价值主张的核心。

| 平台 | 付费支持 | 佣金 | 许可证系统 | 开发者门槛 | 中文流量 | 推荐度 |

### 2.2 各平台详解

#### BuiltByBit（前 MC-Market）— ⭐⭐⭐⭐⭐ 首选

- **流量**：Minecraft 付费插件最大市场，日活买家量大
- **许可证**：内置完整系统 — 买购买后自动生成密钥，插件通过 API 在线验证
- **佣金**：标准 10%，或 $5/月订阅费 → **全免佣金**
- **门槛**：$5 一次性验证费
- **社区**：有买家评价、论坛讨论、技术支持区
- **建议**：付费版首发地

#### Polymart — ⭐⭐⭐⭐ 备选

- **界面**：现代而简洁，用户体验好
- **特性**：支持自动更新推送，买家购买后自动获得新版本
- **佣金**：15%
- **建议**：作为第二渠道扩展曝光

#### SpigotMC Premium — ⚠️ 要慎重

- **流量**：最大，但佣金最高达 **40%**
- **审核**：提交严格，审核周期长
- **建议**：等付费版在 BuiltByBit 有销量和评价后，再考虑上架

### 2.3 免费版保留渠道

即使推出付费版，建议继续在以下平台维护**功能受限的免费版**作为引流：

- **Hangar** — 保持当前免费版
- **GitHub Releases** — 提供开源免费版
- **Modrinth** — 可选的额外曝光

## 3. 商业模式

### 3.1 推荐策略：三层分层模式

```
┌─────────────────────────────────────────────────────────────┐
│  FREE 免费版                                                │
│  发布平台：Hangar / GitHub                                  │
│  功能：                                                     │
│  ├─ 基础白名单管理 (/wl, $a, $r)                            │
│  ├─ 玩家在線查詢 ($l, $w, $h)                               │
│  ├─ 机器人连接状态检查 (/bot)                                │
│  ├─ 引导书 (/guide)                                         │
│  └─ 基础通知（玩家进出服）                                  │
│  许可证：GNU GPL v3                                         │
│  目的：引流、建立口碑                                       │
├─────────────────────────────────────────────────────────────┤
│  PREMIUM 付费版 — $19.99 一次性                              │
│  发布平台：BuiltByBit / Polymart                            │
│  额外功能：                                                  │
│  ├─ TNT 保护系统 (区域白名单、冷却、爆炸告警)              │

│  ├─ 自定义样式系统                                          │
│  └─ 优先技术支持                                            │

│  目的：核心收入来源                                         │
├─────────────────────────────────────────────────────────────┤
```

### 3.2 其他可选模式

| 模式 | 说明 | 适用性 |
|------|------|--------|
| **一次性付费** | 付费下载，终生使用 | ⭐⭐⭐ 主流方案，最易接受 |
| **订阅制** | $4.99-9.99/月，含持续更新+支持 | 适合功能持续迭代的插件 |
| **Freemium** | 基础免费，高级功能付费解锁 | ⭐⭐⭐ 当前推荐 |
| **捐赠制** | 免费使用，自愿赞助 | 收入不稳定，不推荐作为主力 |
| **功能解锁码** | 同一 JAR，购买解锁码激活更多功能 | 技术复杂，易被破解 |
| **SaaS 托管** | 托管式后台 + 机器人 | 适合长远发展，短期不做 |

### 3.3 收入预期估算（保守）

假设 BuiltByBit 一个月的流量：

| 场景 | 月销量 | 单价 | 月收入（扣除佣金） |
|------|--------|------|:------------------:|
| 冷启动（前 3 月） | 5-15 份 | $19.99 | $90-$270 |
| 稳定期（3-12 月） | 15-40 份 | $19.99 | $270-$720 |
| 成熟期（1 年+） | 30-80 份 | $19.99 | $540-$1,440 |

国内渠道追加预估 ¥200-800/月。

### 3.4 收入模型现实检验

上述数字是乐观估计，以下是更保守的现实检验：

#### 对标分析

BuiltByBit 上销量数据（公开可见）显示：
- **头部插件**（如 ItemsAdder、Oraxen、Lands）：月销 100-300 份，但他们已运营 3-5 年，有大量用户评价和社区基础
- **腰部插件**（中等知名度、< 100 评价）：月销 20-60 份
- **尾部插件**（新上架、< 20 评价）：月销 5-15 份

> OrzMC 的合理定位：起步期对标 "尾部"，成熟期对标 "腰部"。头部需要 3 年以上积累。

#### 更现实的收入预测

| 时间段 | 乐观月收入 | 现实月收入 | 保守月收入 |
|:------|:----------:|:----------:|:----------:|
| 0-3 月（冷启动） | $270 | **$60-150** | $20-50 |
| 3-12 月（增长期） | $720 | **$200-400** | $80-150 |
| 12 月+（成熟期） | $1,440 | **$400-800** | $150-300 |

> **坦率地说**：Minecraft 付费插件不是 "暴富" 赛道。即使非常成功的插件，月收入也通常在 $1,000-3,000 区间。将其视为 **"爱好变现 + 持续被动收入"** 比 "创业项目" 更现实。

#### 影响收入的关键变量

| 变量 | 影响 | 可控性 |
|------|:----:|:------:|
| BuiltByBit 产品页质量（截图/视频/文案） | ⭐⭐⭐⭐⭐ | 高 — 全由你控制 |
| 用户评价数量与星级 | ⭐⭐⭐⭐⭐ | 中 — 需靠产品质量 + 售后服务赢得 |
| QQ 机器人稳定性 | ⭐⭐⭐⭐ | 中 — 依赖 NapCatQQ，部分可控 |
| 更新频率 | ⭐⭐⭐ | 高 |
| B站/论坛推广效果 | ⭐⭐⭐ | 高 |
| 竞品动态 | ⭐⭐ | 低 |

## 4. 定价策略

### 4.1 海外定价参考区间

| 插件类型 | 价格区间 | 例子 |
|---------|---------|------|
| 单功能小插件 | $5-10 | 记分板、公告插件 |
| 中大型管理插件 | **$15-25** | Lands ($20), SuperiorSkyblock ($15) |
| 全功能套件 | $30-50 | AdvancedEnchantments ($35), Oraxen ($20) |
| 订阅制 | $5-10/月 | 含持续更新+支持 |

**→ OrzMC 建议海外首价：$19.99（一次性定价）**

### 4.2 国内定价建议（人民币）

| 版本 | 价格 | 说明 |
|------|:----:|------|
| 个人版 | ¥39-69 | 服务器 ≤ 50 人 |
| 商业版 | ¥99-199 | 无人数限制 |
| 技术支持 | ¥299+ | 含配置指导 + 定制开发 |

### 4.3 定价原则

- **首发不宜过高**：初期以低价或折扣积累用户评价（Launch 折扣 -30%）
- **留涨价空间**：定价 $19.99，未来功能扩充后可涨至 $24.99-29.99
- **区分明显**：免费版与付费版功能差异要清晰，让用户觉得付费值得
- **捆绑折扣**：考虑 "买 2 份减 10%" 等批量优惠（大型服务器可能需要多服授权）

## 5. 防盗版方案

付费插件的核心挑战是盗版。建议分层防护：

### 5.1 第一层：平台内置验证（必需）

BuiltByBit 和 Polymart 都提供内置许可证系统：
- 买家购买后收到唯一激活密钥
- 插件启动时通过 HTTP API 向平台验证密钥有效性
- 平台已拦截大部分简单盗版
- **投入：零开发成本**

### 5.2 第二层：自有验证服务器（推荐）

```
插件启动 → 读取许可证密钥 → HTTPS 请求你的验证服务器
  → 服务器解密校验签名
  → 返回：有效 / 无效 / 超过绑定数
  → 无效响应 → 插件行为：

       ┌─ 1 小时内：仅在控制台输出警告
       ├─ 1-24 小时：每次管理员 OP 登录时提醒
       ├─ 24-72 小时：禁用高级功能（保留基础功能保服务器）
       └─ 72 小时+：完全禁用但保留 /orz 帮助命令
```

**实现方案选择：**

| 方案 | 难度 | 成本 | 推荐度 |
|------|:----:|:----:|:------:|
| 自建轻量 API（Java/Sprint Boot） | 中 | 服务器费用 | ⭐⭐⭐⭐ |
| 自建轻量 API（Node.js/Cloudflare Worker） | 低 | 几乎免费 | ⭐⭐⭐⭐⭐ |
| Lunr | 低 | 免费起步 | ⭐⭐⭐ |
| KeyAuth | 低 | 免费起步 | ⭐⭐⭐ |

**独立验证服务器的优势**：即使平台方被攻破，你的自有验证仍然安全。

### 5.3 第三层：代码混淆（建议）

使用 **ProGuard** 或 **yGuard** 在构建时混淆代码：

```kotlin
// build.gradle.kts 示例 — ProGuard 集成
plugins {
    id("com.guardsquare.proguard") version "7.x"
}

tasks.register<ProguardTask>("obfuscate") {
    configuration("proguard.pro")
    injars(tasks.shadowJar.flatMap { it.archiveFile })
    outjars(layout.buildDirectory.file("libs/orzmc-premium-obfuscated.jar"))
    // libjars 包含 PaperAPI 等公共库（不混淆）
}
```

**关键规则：**
- ✅ 混淆你的业务逻辑包（`com.jokerhub.paper.plugin.orzmc.features.*`）
- ✅ 混淆基础设施包（`com.jokerhub.paper.plugin.orzmc.infra.*`）
- ❌ **不要混淆** PaperAPI 的公开类和方法
- ❌ **不要混淆** 插件入口类（`OrzMC`）和 `plugin.yml` 中的引用

### 5.4 第四层：服务器绑定（可选）

- 每个许可证绑定最多 3 个服务器的 IP 或玩家 UUID
- 防止一份购买被无限服务器使用
- 用户可在后台管理绑定（解绑/更换）

### 5.5 对抗盗版的底线思维

> **不存在 100% 防破解的方案。** Someone will always crack it.
>
> 目标不是让破解不可能，而是让：
> 1. 破解成本 > 购买成本
> 2. 破解版本不稳定（加校验、加延迟、加随机检查）
> 3. 普通人找不到/不敢用破解版

## 6. 中国市场策略

### 6.1 为什么中国市场是核心突破口


## 7. 法律与许可

### 7.1 当前状态

此仓库已于 2026-07-03 添加 **GNU GPL v3** 许可证文件（`LICENSE`）。
- ✅ 免费版用户明确拥有使用、修改、分发的权利
- ✅ 版权人（你）保留双许可（dual-licensing）的权利
- ✅ 你可以同时以 GPLv3 发布免费版、以专有许可发布付费版
- ⚠️ GPLv3 的 copyleft 条款要求衍生作品也必须开源，这恰好保护了你的付费版（见下文）

### 7.2 推荐许可策略

| 版本 | 许可证 | 说明 |
|------|--------|------|
| GitHub/Hangar 免费版 | **GNU GPL v3** | 可自由使用、修改、分发，但必须开源 |
| Premium 付费版 | **闭源**（仅二进制 JAR） | 通过购买获得使用许可 |
| Enterprise 企业版 | **自定义商业许可** | 附源代码查看权，但无再分发权 |

**为什么免费版选 GPL v3 而不是 MIT？**
- MIT 允许任意使用，包括别人把你的代码打包成商业产品卖
- GPL v3 要求衍生作品也必须开源，有效保护你的商业版代码
- GPL v3 的传染性也为付费版提供了竞争优势（想闭源集成 → 买商业版）

### 7.3 GPLv3 双许可策略深度分析

GPLv3 不是商业化的障碍——恰恰相反，它是开源商业模式中最成熟的策略之一。

#### 法律基础

GPLv3 第 4 条明确指出：**"你可以为每一份副本收取任何价格或免费提供，你也可以为提供支持或质保而收取费用。"** 这意味着：
- 你可以卖 GPL 软件赚钱（但购买者获得源码后有权自由再分发）
- **但是**，你作为版权持有人（copyright holder），不受 GPL 条款约束——你可以同时以不同许可发布相同代码
- 这就是 **"双许可"（dual-licensing）** 模式：免费版 GPLv3（必须开源） + 付费版专有许可（可闭源）

#### 在 OrzMC 中的具体实施

```
同一代码库
    ├─ free 分支：GPLv3 发布 → 包含基础功能源码
    │     ↑ 所有 external contributor 的代码在此，受 GPLv3 保护
    │
    └─ premium 分支：专有许可发布 → 仅二进制 JAR
          ↑ 仅使用你作为 copyright holder 的代码
          ↑ 不包含任何 third-party GPL 代码
```

**关键规则：**
| 规则 | 说明 |
|------|------|
| 你是唯一 copyright holder | 目前所有代码都是你写的——不存在版权冲突 |
| 接受外部 PR 时必须要求 CLA | Contributor License Agreement，让贡献者将其代码版权授予你，或允许你以专有许可再发布 |
| 付费版不可包含 GPL 依赖 | 如果某个库/补丁仅在 GPL 下提供且你不持有其版权，不得用于付费版 |
| orzmc-api 子模块 | 如果用作公开 SDK，其许可需要单独考虑（建议同样 GPLv3，或考虑 LGPL 以允许商业集成） |

#### 为什么双许可比纯闭源更优

| 维度 | 纯闭源 | GPLv3 双许可 |
|------|:------:|:------------:|
| 社区贡献 | 几乎为零 | 可接受外部 PR（走 CLA） |
| 品牌信任 | 低 | 高——开源代码可审查 |
| 竞争防御 | 靠混淆/法律 | 靠 GPL 传染性 + 品牌 |
| 用户获取 | 全靠付费 | 免费版引流 → 付费版转化 |
| 合规成本 | 低 | 中（需 CLA 流程 + 版权审计） |
| 被盗版风险 | 相同 | 相同（付费版 JAR 同样被逆向） |
| Hangar/Modrinth 发布 | 不可（仅免费） | ✅ 可发布免费版 |

#### 各平台对 GPL 的兼容性

| 平台 | GPL 免费版 | 专有付费版 | 备注 |
|------|:----------:|:----------:|------|
| **Hangar** | ✅ 原生支持 | ❌ 仅免费 | 可展示"升级 Premium"链接 |
| **Modrinth** | ✅ 原生支持 | ❌ 仅免费 | 同 Hangar |
| **GitHub** | ✅ | ❌ | README 引流至付费版 |
| **BuiltByBit** | ❌（无需放免费版） | ✅ | 付费版主战场 |
| **Polymart** | ❌（无需放免费版） | ✅ | 第二付费渠道 |

#### 常见误区澄清

> **误区 1**："GPL 软件不能卖钱"
> → **错误。** GPL 明确允许收费。问题在于购买者获得源码后可以免费再分发，所以"卖 GPL 副本"作为商业模式不现实。但"卖专有许可 + GPL 免费版引流"完全可行。

> **误区 2**："用了 GPL 就必须公开全部代码"
> → **部分正确。** GPL 的 copyleft 条款要求分发二进制时必须同时提供源码。但这只约束被许可方（licensee），不约束版权持有人（licensor）。你是版权持有人，不受此限。

> **误区 3**："GPL 代码不能用于商业产品"
> → **错误。** 只要你不将别人的 GPL 代码混入你的专有产品，你的专有代码与 GPL 代码是独立作品。作为唯一版权持有人，你写的代码不是"别人的 GPL 代码"——你拥有它，可以随意再许可。

## 10. 风险评估

| **市场太小，收入不及预期** | 中 | 高 | Minecraft 付费插件市场确实小众。降低期望：把收入当副业补充而非主要收入来源。先从 BuiltByBit 零成本起步，验证需求后再加大投入。 |
| **竞品降价或推出免费替代** | 低 | 中 | 你的核心差异化（QQ 集成）竞品难以复制——海外开发者不理解中国市场。保持功能迭代领先。 |
| **Minecraft 版本大更新导致插件失效** | 中 | 中 | 已有成熟 CI/CD，持续跟进 Paper 更新即可。这是所有 MC 插件共同面临的风险。 |
| **国内用户付费意愿低** | 高 | 低 | 国内定价已有意调低（¥39-69 vs 海外 $19.99）。同时国内有 QQ 群直销渠道，零佣金补充收入。 |

### 10.2 技术风险

|------|:----:|:----:|---------|
| **NapCatQQ/OneBot 协议变更** | 中 | 高 | QQ Bot 是核心差异化功能。关注上游项目动态，保持与社区的联系。考虑长期自研 QQ Bot 适配层。 |

| **用户支持占用过多时间** | 高 | 中 | 写好文档和 FAQ。设置支持边界：免费版仅 GitHub Issues，付费版优先响应。考虑 Discord 社区互助 + 知识库自助。 |
| **退款/纠纷** | 低 | 低 | BuiltByBit 有成熟的退款机制。自建渠道制定明确的退款政策（如 7 天无理由）。 |
| **付款平台政策变更** | 低 | 中 | BuiltByBit 和 Polymart 长期运营，风险低。国内渠道多平台分散风险。 |
| **法律合规风险** | 低 | 中 | GPLv3 已是主流开源许可，合规简单。付费版 EULA 可使用模板。Verification Server 涉及数据收集需隐私条款。 |

### 10.4 风险评估总结

| 风险等级 | 行动 |
|:--------:|------|
| 🟢 **低风险** | 常规监控，无需特殊应对 |
| 🟡 **中风险** | 制定缓解计划，定期回顾 |
| 🔴 **高风险** | 需要主动缓解措施，作为 Phase 1 优先级 |

**最需要关注的两个风险：**
1. **市场太小** — 这是最现实的顾虑。建议以 "验证需求" 而非 "追求收入" 的心态启动：先在 BuiltByBit 上架，花 $5 验证费试水 3 个月，看真实数据再决定是否持续投入。
2. **NapCatQQ 协议稳定性** — 直接影响核心竞争力。保持对上游项目的关注，必要时有能力自行维护适配层。
